# Claude agents — context for AI agents working here

This directory (`claude/`) holds custom Claude Code subagents. Each agent is a folder: `<name>/<name>.md` (Markdown + YAML frontmatter) plus `<name>/kaizen/{plan,history}.md`. **Skills no longer live here** — they were unified into the repo-root [`skills/`](../skills/) home (see [`skills/README.md`](../skills/README.md)); cobb's `agent-maintenance` and `agent-standards` skills are there. See [`README.md`](./README.md) for the human-facing agent catalog.

## Agents

- **teco** — technical coordinator / tech lead; decomposes a multi-step goal into a sequenced work breakdown and **delegates each unit to the right specialist** (architect, coder, tdd-engineer, qa-engineer, graph-dba, devops, cobb; Explore/Plan built-ins) via the `Agent` tool — independent delegations issued in parallel — then integrates + verifies (running the project's suites itself is in-bounds; acceptance-level verification routes to `qa-engineer`, environment blockers to `devops`, with a defect→fix→re-run loop when QA finds problems). **Hybrid:** drives execution itself but returns to the user at genuine decision points/blockers (subagents can't use `AskUserQuestion`). Coordinates only — no designing or coding; `Write`/`Edit` are scoped to the coordination doc (Write to create, Edit to revise in place; convention `<component>/docs/plans/<slug>-coordination.md`, co-located with the architect's plan), never source/tests/config — **harness-enforced**: a subagent-scoped `PreToolUse` hook (`teco/hooks/guard-coordination-doc-writes.sh`, matcher `Write|Edit`) escalates any target outside `docs/plans/` (or the `/tmp` scratchpad) to the human. Briefs to specialists must be self-contained (they don't share teco's context) and remind delegates to return blockers/questions as their deliverable; document artifacts are handed off **by path** — the architect's plan at `docs/plans/<slug>.md` (and likewise the qa-engineer's test plan/report) is passed onward as a path to read, never paraphrased into the brief. Source: `teco/teco.md`. Kaizen: `teco/kaizen/`. (Verified 2026-06-20: subagents may use the `Agent` tool, so an orchestrator subagent is supported.)
- **architect** — software architect; investigates the codebase, weighs trade-offs, and produces a step-by-step implementation plan/spec (goal & scope, findings, design & rationale, ordered steps, test strategy, risks). **Default deliverable is a plan document** at `<component>/docs/plans/<slug>.md` (repo-root `docs/plans/` for cross-component work) — it returns the path + a "ready to implement" summary, so orchestrators hand off the path losslessly; inline only for quick assessments. **Read-only on code, harness-enforced** — `Write`/`Edit` are for the plan/design document only (Write to create, Edit to revise), and a subagent-scoped `PreToolUse` hook (`architect/hooks/guard-plan-doc-writes.sh`, matcher `Write|Edit`) escalates any target outside `docs/plans/` (or the `/tmp` scratchpad) to the human; `Bash` stays prompt-guarded (investigation only); no `NotebookEdit`, never touches source/tests/config. Subagent-context aware: treats its brief as its entire input and, when design-changing ambiguity hits, returns findings + open questions as the deliverable instead of trying to ask mid-run. The planning half of an **architect→coder handoff**; the plan is written to stand alone for an isolated-context implementer. Use for designs, approaches, impact analysis, or a plan before code. Source: `architect/architect.md`. Kaizen: `architect/kaizen/`.
- **coder** — software engineer who implements an approved plan/spec end-to-end: clean, idiomatic, well-tested code matching repo conventions, suite kept green, honest run reporting. Inherits all tools. The implementation half of the architect→coder handoff — an architect plan arrives as a document path (`<component>/docs/plans/<slug>.md`); the coder reads the file as its source of truth. **Distinct from `tdd-engineer`:** plan-driven and pragmatic about testing (tests alongside, not test-first); routing between the two is by **efficiency** — a detailed plan/spec ready to execute → `coder`; a bug fix, safety-net refactor, test-focused work, or clear-contract feature where test-first is the efficient path → `tdd-engineer`. Subagent-aware (returns environment blockers / destructive-op requests as its result when delegated, e.g. by teco). Use when building from a ready plan or clear task. Source: `coder/coder.md`. Kaizen: `coder/kaizen/`.
- **cobb** — agentic-development expert (Claude Code, Kiro, OpenCode agent formats + cross-tool standards). Use for designing/authoring/reviewing/porting/debugging agents, skills, steering docs, hooks, system prompts. Source: `cobb/cobb.md`. Kaizen: `cobb/kaizen/`. Testing standards: `cobb/TESTING.md`. Relies on two skills in the repo-root [`skills/`](../skills/) home: **`agent-maintenance`** for its kaizen/documentation/drift-audit procedures **and the team-coherence certification pass** (inter-agent rosters, handoff symmetry, subagent-awareness, hook enforcement parity; deterministic half scripted in `claude/scripts/audit-team.sh`), and **`agent-standards`** for the perishable per-tool reference specifics (frontmatter fields, paths, inclusion modes). The resident prompt keeps only the mandate + stable mental models and points at the skills.
- **tdd-engineer** — implements features/fixes strictly via TDD (red → green → refactor). Accepts an architect plan as a document path (`<component>/docs/plans/<slug>.md`) and reads the file as its source of truth — the plan's test-strategy section drives the red→green sequence; subagent-aware (returns questions/blockers as its result when delegated, e.g. by teco). Use where test-first is the efficient path: a bug fix (reproduction test first), a refactor needing a safety net, adding/improving tests, or a feature with a clear up-front behavior contract; a detailed plan ready to execute routes to `coder` instead. Source: `tdd-engineer/tdd-engineer.md`. Kaizen: `tdd-engineer/kaizen/`.
- **qa-engineer** — QA / functional-testing engineer working at the **behavior/acceptance altitude** (the black-box complement to `tdd-engineer`'s unit-level TDD). Four-phase loop: **reason** a risk-based test strategy → **write** it to a versioned test plan following the component's doc convention (`docs/test-plans/<kebab>.md`, kebab-case per feature) → **execute** it three ways (authors automated functional/acceptance/contract/e2e tests, runs existing suites/scripts like `./scripts/test_queries.sh` + `pytest`, **and** drives the running app black-box via curl/HTTP/MCP/scripts) → **report** to `docs/test-reports/<kebab>-report.md` with results table, reproducible defects (severity by user impact), coverage/gaps, and feedback. Evidence-over-assertion (shows real output, never fabricates a pass); establishes a green baseline first and reports blockers rather than piling on red; does **not** fix the code under test unless asked (defers to `coder`/`tdd-engineer`) and never mutates the shared FalkorDB env without approval. Inherits all tools (needs Write/Edit/Bash to author tests, run suites, drive apps). Subagent-aware: when delegated (e.g. by teco) it returns sharp questions / environment-approval requests as its result instead of trying to ask mid-run. Use for a test strategy/plan, functional/acceptance/integration/e2e/exploratory testing, a QA pass, or a written test report. Source: `qa-engineer/qa-engineer.md`. Kaizen: `qa-engineer/kaizen/`.
- **graph-dba** — graph database administrator & data architect specialized in **FalkorDB** (Redis-module, GraphBLAS sparse-matrix engine; RedisGraph successor; GraphRAG-focused), fluent in the wider LPG world (Neo4j, openCypher, GQL) for porting. Use for graph data modeling, writing/optimizing FalkorDB Cypher, vector/full-text indexing & constraints, deployment (RAM sizing, persistence, replication, Redis Cluster), query tuning (`GRAPH.PROFILE`), bulk ingestion/migration, GraphRAG/knowledge-graph layers, and ops. Carries a **live-verified quirks knowledge base** for this lab's FalkorDB edge build (`graph-dba/falkordb-quirks.md`) — a dated, perishable resource file the prompt points at (kept out of the always-on prompt); re-verify on any tagged-release upgrade. Subagent-aware (returns its sharp design question as its result when delegated, e.g. by teco). Source: `graph-dba/graph-dba.md`. Kaizen: `graph-dba/kaizen/`.

- **devops** — DevOps / platform engineer; owns **environments, containerization, and the delivery lifecycle** — but is **project-agnostic** (it's symlinked into `~/.claude/agents/`, i.e. **user scope**, so it runs in *every* project, not just this repo). **Its first discipline is orientation:** in any repo it reads the project's README, `AGENTS.md`/`CLAUDE.md`, `docs/`, and infra files (Dockerfiles, Compose, CI configs, dependency manifests, scripts, `.env.example`) plus live state (`docker ps`, `git ls-files`) to build an *infra brief* before acting — it never assumes a stack. (It knows the `CLAUDE.md`/`AGENTS.md` memory hierarchy auto-loads but README/`docs/` do **not**, so it actively reads them.) Skills: Docker/Compose, reproducible dev environments, cross-ecosystem dependency/venv management (pip/uv/poetry, npm, cargo, go mod…), `.env` & **secrets hygiene**, robust Bash automation, CI/CD, deploy/release, observability. **Autonomy = build + guarded ops:** authors/edits infra files and runs build/inspect commands freely, but **stops for explicit approval before destructive/shared-state ops** (`docker volume rm`, `system prune`, evicting a running container, or any write against a shared/live datastore). Inherits all tools, but the destructive-op guardrail is **harness-enforced**, not just prompt text: a subagent-scoped `PreToolUse` hook (`devops/hooks/guard-destructive-ops.sh`, wired in its frontmatter) intercepts `docker volume rm`/`prune`, `system prune`, `docker rm -f`, `compose down -v`, and Redis/FalkorDB `FLUSHALL`/`FLUSHDB`/`GRAPH.DELETE` and escalates them to the human for approval (`permissionDecision: "ask"`). Within **this** repo the shared service it must guard is the FalkorDB container (`falkordb/falkordb:edge`, 6379/3000, volume `falkordb-data`; the two `start_falkordb.sh` scripts both bind 6379). Boundaries: DB/data-model/Cypher/ACL design → `graph-dba` (or the project's DBA), app features → `coder`/`tdd-engineer`, test strategy → `qa-engineer`, agent authoring → `cobb`. Source: `devops/devops.md`. Kaizen: `devops/kaizen/`.

## Maintenance rules

- Adding/editing/renaming/removing an agent → update the agent source, its `kaizen/{plan,history}.md`, this file, and `README.md` in the same change.
- Skills live in the repo-root [`skills/`](../skills/) home, not here. Their catalog is [`skills/README.md`](../skills/README.md); cobb's kaizen logs changes to `agent-maintenance`/`agent-standards`.
- Don't paste full system prompts here — point to the source file.
