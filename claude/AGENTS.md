# Claude agents — context for AI agents working here

This directory (`claude/`) holds custom Claude Code subagents. Each agent is a folder: `<name>/<name>.md` (Markdown + YAML frontmatter) plus `<name>/kaizen/{plan,history}.md`. **Skills no longer live here** — they were unified into the repo-root [`skills/`](../skills/) home (see [`skills/README.md`](../skills/README.md)); cobb's `agent-maintenance` and `agent-standards` skills are there. See [`README.md`](./README.md) for the human-facing agent catalog.

## Agents

- **teco** — technical coordinator / tech lead; decomposes a multi-step goal into a sequenced work breakdown and **delegates each unit to the right specialist** (architect, coder, tdd-engineer, graph-dba, cobb; Explore/Plan built-ins) via the `Agent` tool, then integrates + verifies. **Hybrid:** drives execution itself but returns to the user at genuine decision points/blockers (subagents can't use `AskUserQuestion`). Coordinates only — no designing or coding; `Write` is for the coordination doc, no `Edit`. Briefs to specialists must be self-contained (they don't share teco's context). Source: `teco/teco.md`. Kaizen: `teco/kaizen/`. (Verified 2026-06-20: subagents may use the `Agent` tool, so an orchestrator subagent is supported.)
- **architect** — software architect; investigates the codebase, weighs trade-offs, and produces a step-by-step implementation plan/spec (goal & scope, findings, design & rationale, ordered steps, test strategy, risks). **Read-only on code** — `Write`/`Edit` are for the plan/design document only (Write to create, Edit to revise); no `NotebookEdit`, and never touches source/tests/config. The planning half of an **architect→coder handoff**; the plan is written to stand alone for an isolated-context implementer. Use for designs, approaches, impact analysis, or a plan before code. Source: `architect/architect.md`. Kaizen: `architect/kaizen/`.
- **coder** — software engineer who implements an approved plan/spec end-to-end: clean, idiomatic, well-tested code matching repo conventions, suite kept green, honest run reporting. Inherits all tools. The implementation half of the architect→coder handoff. **Distinct from `tdd-engineer`:** plan-driven and pragmatic about testing, not strictly test-first — route strict red→green→refactor work to `tdd-engineer`. Use when building from a ready plan or clear task. Source: `coder/coder.md`. Kaizen: `coder/kaizen/`.
- **cobb** — agentic-development expert (Claude Code, Kiro, OpenCode agent formats + cross-tool standards). Use for designing/authoring/reviewing/porting/debugging agents, skills, steering docs, hooks, system prompts. Source: `cobb/cobb.md`. Kaizen: `cobb/kaizen/`. Testing standards: `cobb/TESTING.md`. Relies on two skills in the repo-root [`skills/`](../skills/) home: **`agent-maintenance`** for its kaizen/documentation/drift-audit procedures, and **`agent-standards`** for the perishable per-tool reference specifics (frontmatter fields, paths, inclusion modes). The resident prompt keeps only the mandate + stable mental models and points at the skills.
- **dra-claudia** — médica de homeopatia/medicina alternativa; mantém prontuários em markdown. Use para perguntas de saúde/sintomas/tratamentos e registro/consulta de histórico clínico (não substitui consulta presencial). Source: `dra-claudia/dra-claudia.md`. Kaizen: `dra-claudia/kaizen/`.
- **saul** — assessor jurídico (foro de SP) de direito **civil e penal**, foco aprofundado em **direito condominial** (condomínio edilício: CC arts. 1.331–1.358-A, Lei 4.591/64, convenção/regimento, cotas como título executivo CPC 784,X, multas 1.336/1.337). Monta dossiês em markdown por caso (`$AGENT_WORKDIR/saul/dossies/{cliente}/{caso}.md` — resolve `$AGENT_WORKDIR` via `Bash` no início da sessão), traça estratégia e redige minutas (sempre tarjadas **RASCUNHO** — revisão por advogado OAB). Disciplina de fonte: cita os dispositivos, busca vigência atual na web (fontes oficiais) antes de afirmar, e rotula 🟢 verificado / 🟡 tese / 🔴 a confirmar — nunca trata inferência como fato nem inventa jurisprudência. Áreas conexas (Consumidor/Família/Trabalhista) ativáveis sob demanda. `tools: Read, Write, Edit, Bash, WebFetch, WebSearch`. Source: `saul/saul.md`. Kaizen: `saul/kaizen/`.
- **tdd-engineer** — implements features/fixes strictly via TDD (red → green → refactor). Use for implementing a feature, fixing a bug, refactoring with a safety net, or adding/improving tests. Source: `tdd-engineer/tdd-engineer.md`. Kaizen: `tdd-engineer/kaizen/`.
- **qa-engineer** — QA / functional-testing engineer working at the **behavior/acceptance altitude** (the black-box complement to `tdd-engineer`'s unit-level TDD). Four-phase loop: **reason** a risk-based test strategy → **write** it to a versioned test plan following the component's doc convention (`docs/test-plans/<kebab>.md`, kebab-case per feature) → **execute** it three ways (authors automated functional/acceptance/contract/e2e tests, runs existing suites/scripts like `./scripts/test_queries.sh` + `pytest`, **and** drives the running app black-box via curl/HTTP/MCP/scripts) → **report** to `docs/test-reports/<kebab>-report.md` with results table, reproducible defects (severity by user impact), coverage/gaps, and feedback. Evidence-over-assertion (shows real output, never fabricates a pass); establishes a green baseline first and reports blockers rather than piling on red; does **not** fix the code under test unless asked (defers to `coder`/`tdd-engineer`) and never mutates the shared FalkorDB env without approval. Inherits all tools (needs Write/Edit/Bash to author tests, run suites, drive apps). Use for a test strategy/plan, functional/acceptance/integration/e2e/exploratory testing, a QA pass, or a written test report. Source: `qa-engineer/qa-engineer.md`. Kaizen: `qa-engineer/kaizen/`.
- **graph-dba** — graph database administrator & data architect specialized in **FalkorDB** (Redis-module, GraphBLAS sparse-matrix engine; RedisGraph successor; GraphRAG-focused), fluent in the wider LPG world (Neo4j, openCypher, GQL) for porting. Use for graph data modeling, writing/optimizing FalkorDB Cypher, vector/full-text indexing & constraints, deployment (RAM sizing, persistence, replication, Redis Cluster), query tuning (`GRAPH.PROFILE`), bulk ingestion/migration, GraphRAG/knowledge-graph layers, and ops. Carries a **live-verified quirks knowledge base** for this lab's FalkorDB edge build (`graph-dba/falkordb-quirks.md`) — a dated, perishable resource file the prompt points at (kept out of the always-on prompt); re-verify on any tagged-release upgrade. Source: `graph-dba/graph-dba.md`. Kaizen: `graph-dba/kaizen/`.

- **devops** — DevOps / platform engineer; owns **environments, containerization, and the delivery lifecycle** — but is **project-agnostic** (it's symlinked into `~/.claude/agents/`, i.e. **user scope**, so it runs in *every* project, not just this repo). **Its first discipline is orientation:** in any repo it reads the project's README, `AGENTS.md`/`CLAUDE.md`, `docs/`, and infra files (Dockerfiles, Compose, CI configs, dependency manifests, scripts, `.env.example`) plus live state (`docker ps`, `git ls-files`) to build an *infra brief* before acting — it never assumes a stack. (It knows the `CLAUDE.md`/`AGENTS.md` memory hierarchy auto-loads but README/`docs/` do **not**, so it actively reads them.) Skills: Docker/Compose, reproducible dev environments, cross-ecosystem dependency/venv management (pip/uv/poetry, npm, cargo, go mod…), `.env` & **secrets hygiene**, robust Bash automation, CI/CD, deploy/release, observability. **Autonomy = build + guarded ops:** authors/edits infra files and runs build/inspect commands freely, but **stops for explicit approval before destructive/shared-state ops** (`docker volume rm`, `system prune`, evicting a running container, or any write against a shared/live datastore). Inherits all tools, but the destructive-op guardrail is **harness-enforced**, not just prompt text: a subagent-scoped `PreToolUse` hook (`devops/hooks/guard-destructive-ops.sh`, wired in its frontmatter) intercepts `docker volume rm`/`prune`, `system prune`, `docker rm -f`, `compose down -v`, and Redis/FalkorDB `FLUSHALL`/`FLUSHDB`/`GRAPH.DELETE` and escalates them to the human for approval (`permissionDecision: "ask"`). Within **this** repo the shared service it must guard is the FalkorDB container (`falkordb/falkordb:edge`, 6379/3000, volume `falkordb-data`; the two `start_falkordb.sh` scripts both bind 6379). Boundaries: DB/data-model/Cypher/ACL design → `graph-dba` (or the project's DBA), app features → `coder`/`tdd-engineer`, test strategy → `qa-engineer`, agent authoring → `cobb`. Source: `devops/devops.md`. Kaizen: `devops/kaizen/`.

All ten use `model: opus`.

## Maintenance rules

- Adding/editing/renaming/removing an agent → update the agent source, its `kaizen/{plan,history}.md`, this file, and `README.md` in the same change.
- Skills live in the repo-root [`skills/`](../skills/) home, not here. Their catalog is [`skills/README.md`](../skills/README.md); cobb's kaizen logs changes to `agent-maintenance`/`agent-standards`.
- Don't paste full system prompts here — point to the source file.
