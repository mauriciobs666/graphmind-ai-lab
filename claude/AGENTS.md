# Claude agents вЂ” context for AI agents working here

This directory (`claude/`) holds custom Claude Code subagents. Each agent is a folder: `<name>/<name>.md` (Markdown + YAML frontmatter) plus `<name>/kaizen/{plan,history}.md`. **Skills no longer live here** вЂ” they were unified into the repo-root [`skills/`](../skills/) home (see [`skills/README.md`](../skills/README.md)); cobb's `agent-maintenance` and `agent-standards` skills are there. See [`README.md`](./README.md) for the human-facing agent catalog.

## Agents

- **teco** вЂ” technical coordinator / tech lead; decomposes a multi-step goal into a sequenced work breakdown and **delegates each unit to the right specialist** (architect, coder, tdd-engineer, graph-dba, cobb; Explore/Plan built-ins) via the `Agent` tool, then integrates + verifies. **Hybrid:** drives execution itself but returns to the user at genuine decision points/blockers (subagents can't use `AskUserQuestion`). Coordinates only вЂ” no designing or coding; `Write`/`Edit` are scoped to the coordination doc (Write to create, Edit to revise in place; convention `<component>/docs/plans/<slug>-coordination.md`, co-located with the architect's plan), never source/tests/config. Briefs to specialists must be self-contained (they don't share teco's context); document artifacts are handed off **by path** вЂ” the architect's plan at `docs/plans/<slug>.md` is passed to the implementer as a path to read, never paraphrased into the brief. Source: `teco/teco.md`. Kaizen: `teco/kaizen/`. (Verified 2026-06-20: subagents may use the `Agent` tool, so an orchestrator subagent is supported.)
- **architect** вЂ” software architect; investigates the codebase, weighs trade-offs, and produces a step-by-step implementation plan/spec (goal & scope, findings, design & rationale, ordered steps, test strategy, risks). **Default deliverable is a plan document** at `<component>/docs/plans/<slug>.md` (repo-root `docs/plans/` for cross-component work) вЂ” it returns the path + a "ready to implement" summary, so orchestrators hand off the path losslessly; inline only for quick assessments. **Read-only on code, harness-enforced** вЂ” `Write`/`Edit` are for the plan/design document only (Write to create, Edit to revise), and a subagent-scoped `PreToolUse` hook (`architect/hooks/guard-plan-doc-writes.sh`, matcher `Write|Edit`) escalates any target outside `docs/plans/` (or the `/tmp` scratchpad) to the human; `Bash` stays prompt-guarded (investigation only); no `NotebookEdit`, never touches source/tests/config. Subagent-context aware: treats its brief as its entire input and, when design-changing ambiguity hits, returns findings + open questions as the deliverable instead of trying to ask mid-run. The planning half of an **architectв†’coder handoff**; the plan is written to stand alone for an isolated-context implementer. Use for designs, approaches, impact analysis, or a plan before code. Source: `architect/architect.md`. Kaizen: `architect/kaizen/`.
- **coder** вЂ” software engineer who implements an approved plan/spec end-to-end: clean, idiomatic, well-tested code matching repo conventions, suite kept green, honest run reporting. Inherits all tools. The implementation half of the architectв†’coder handoff вЂ” an architect plan arrives as a document path (`<component>/docs/plans/<slug>.md`); the coder reads the file as its source of truth. **Distinct from `tdd-engineer`:** plan-driven and pragmatic about testing, not strictly test-first вЂ” route strict redв†’greenв†’refactor work to `tdd-engineer`. Use when building from a ready plan or clear task. Source: `coder/coder.md`. Kaizen: `coder/kaizen/`.
- **cobb** вЂ” agentic-development expert (Claude Code, Kiro, OpenCode agent formats + cross-tool standards). Use for designing/authoring/reviewing/porting/debugging agents, skills, steering docs, hooks, system prompts. Source: `cobb/cobb.md`. Kaizen: `cobb/kaizen/`. Testing standards: `cobb/TESTING.md`. Relies on two skills in the repo-root [`skills/`](../skills/) home: **`agent-maintenance`** for its kaizen/documentation/drift-audit procedures, and **`agent-standards`** for the perishable per-tool reference specifics (frontmatter fields, paths, inclusion modes). The resident prompt keeps only the mandate + stable mental models and points at the skills.

- **tdd-engineer** вЂ” implements features/fixes strictly via TDD (red в†’ green в†’ refactor). Use for implementing a feature, fixing a bug, refactoring with a safety net, or adding/improving tests. Source: `tdd-engineer/tdd-engineer.md`. Kaizen: `tdd-engineer/kaizen/`.
- **qa-engineer** вЂ” QA / functional-testing engineer working at the **behavior/acceptance altitude** (the black-box complement to `tdd-engineer`'s unit-level TDD). Four-phase loop: **reason** a risk-based test strategy в†’ **write** it to a versioned test plan following the component's doc convention (`docs/test-plans/<kebab>.md`, kebab-case per feature) в†’ **execute** it three ways (authors automated functional/acceptance/contract/e2e tests, runs existing suites/scripts like `./scripts/test_queries.sh` + `pytest`, **and** drives the running app black-box via curl/HTTP/MCP/scripts) в†’ **report** to `docs/test-reports/<kebab>-report.md` with results table, reproducible defects (severity by user impact), coverage/gaps, and feedback. Evidence-over-assertion (shows real output, never fabricates a pass); establishes a green baseline first and reports blockers rather than piling on red; does **not** fix the code under test unless asked (defers to `coder`/`tdd-engineer`) and never mutates the shared FalkorDB env without approval. Inherits all tools (needs Write/Edit/Bash to author tests, run suites, drive apps). Use for a test strategy/plan, functional/acceptance/integration/e2e/exploratory testing, a QA pass, or a written test report. Source: `qa-engineer/qa-engineer.md`. Kaizen: `qa-engineer/kaizen/`.
- **graph-dba** вЂ” graph database administrator & data architect specialized in **FalkorDB** (Redis-module, GraphBLAS sparse-matrix engine; RedisGraph successor; GraphRAG-focused), fluent in the wider LPG world (Neo4j, openCypher, GQL) for porting. Use for graph data modeling, writing/optimizing FalkorDB Cypher, vector/full-text indexing & constraints, deployment (RAM sizing, persistence, replication, Redis Cluster), query tuning (`GRAPH.PROFILE`), bulk ingestion/migration, GraphRAG/knowledge-graph layers, and ops. Carries a **live-verified quirks knowledge base** for this lab's FalkorDB edge build (`graph-dba/falkordb-quirks.md`) вЂ” a dated, perishable resource file the prompt points at (kept out of the always-on prompt); re-verify on any tagged-release upgrade. Source: `graph-dba/graph-dba.md`. Kaizen: `graph-dba/kaizen/`.

- **devops** вЂ” DevOps / platform engineer; owns **environments, containerization, and the delivery lifecycle** вЂ” but is **project-agnostic** (it's symlinked into `~/.claude/agents/`, i.e. **user scope**, so it runs in *every* project, not just this repo). **Its first discipline is orientation:** in any repo it reads the project's README, `AGENTS.md`/`CLAUDE.md`, `docs/`, and infra files (Dockerfiles, Compose, CI configs, dependency manifests, scripts, `.env.example`) plus live state (`docker ps`, `git ls-files`) to build an *infra brief* before acting вЂ” it never assumes a stack. (It knows the `CLAUDE.md`/`AGENTS.md` memory hierarchy auto-loads but README/`docs/` do **not**, so it actively reads them.) Skills: Docker/Compose, reproducible dev environments, cross-ecosystem dependency/venv management (pip/uv/poetry, npm, cargo, go modвЂ¦), `.env` & **secrets hygiene**, robust Bash automation, CI/CD, deploy/release, observability. **Autonomy = build + guarded ops:** authors/edits infra files and runs build/inspect commands freely, but **stops for explicit approval before destructive/shared-state ops** (`docker volume rm`, `system prune`, evicting a running container, or any write against a shared/live datastore). Inherits all tools, but the destructive-op guardrail is **harness-enforced**, not just prompt text: a subagent-scoped `PreToolUse` hook (`devops/hooks/guard-destructive-ops.sh`, wired in its frontmatter) intercepts `docker volume rm`/`prune`, `system prune`, `docker rm -f`, `compose down -v`, and Redis/FalkorDB `FLUSHALL`/`FLUSHDB`/`GRAPH.DELETE` and escalates them to the human for approval (`permissionDecision: "ask"`). Within **this** repo the shared service it must guard is the FalkorDB container (`falkordb/falkordb:edge`, 6379/3000, volume `falkordb-data`; the two `start_falkordb.sh` scripts both bind 6379). Boundaries: DB/data-model/Cypher/ACL design в†’ `graph-dba` (or the project's DBA), app features в†’ `coder`/`tdd-engineer`, test strategy в†’ `qa-engineer`, agent authoring в†’ `cobb`. Source: `devops/devops.md`. Kaizen: `devops/kaizen/`.

- Skills live in the repo-root [`skills/`](../skills/) home, not here. Their catalog is [`skills/README.md`](../skills/README.md); cobb's kaizen logs changes to `agent-maintenance`/`agent-standards`.
- Don't paste full system prompts here вЂ” point to the source file.
